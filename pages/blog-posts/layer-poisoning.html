Layer Poisoning, l'art de compromettre Docker par ses fondations

On nous présente souvent les conteneurs Docker comme des environnements isolés et immuables. Pourtant, il existe une technique d'attaque appelée Layer Poisoning. Elle permet de compromettre un conteneur en modifiant silencieusement ses couches de système de fichiers directement sur l'hôte. Retour sur une manipulation qui casse le mythe de l'isolation parfaite.

1. Le concept : Empoisonner la source

Le "Layer Poisoning" consiste à injecter du code malveillant dans les snapshots de stockage de Docker (gérés par containerd). Ce n'est pas un bug de Docker en soi, mais une exploitation directe de son architecture. Si un attaquant parvient à obtenir un accès root sur l'hôte, il peut modifier les fichiers "sources" que le conteneur utilise pour démarrer. Résultat ? Une backdoor persistante, invisible pour les outils de scan classiques.

2. Les coulisses : Docker et OverlayFS

Pour comprendre l'attaque, il faut comprendre OverlayFS. Docker empile les couches d'images pour former le système de fichiers final du conteneur. Ce mécanisme repose sur trois piliers :

  • Lowerdir : Les couches de l'image de base (ex: Alpine), normalement en lecture seule.
  • Upperdir : La couche d'écriture propre au conteneur (où vont vos modifications).
  • Merged : La vue fusionnée que l'utilisateur voit à l'intérieur du conteneur.

En temps normal, le lowerdir est protégé. Mais sur l'hôte, ces fichiers sont stockés dans /var/lib/containerd/.../snapshots/. En modifiant ces snapshots, on modifie l'ADN même du conteneur.

4. Pourquoi cette attaque est redoutable ?

Ce qui rend le Layer Poisoning particulièrement vicieux, c'est sa discrétion. Si vous lancez un docker diff, Docker ne verra aucune modification. Pourquoi ? Parce que la modification a eu lieu dans le Lowerdir (l'image de base) et non dans la couche d'écriture du conteneur. De plus, tout nouveau conteneur créé à partir de cette image locale sera nativement infecté.

5. Conclusion et Défense

Cette démonstration prouve qu'une isolation logicielle ne vaut rien si l'intégrité de l'hôte est compromise. Pour dormir tranquille, voici quelques pistes :

  • Principe du moindre privilège : Restreindre l'accès aux dossiers de containerd/docker sur l'hôte.
  • Immuabilité : Utiliser des systèmes de fichiers en lecture seule (--read-only) pour limiter l'impact d'une intrusion.
  • Vérification : Mettre en place des outils de monitoring d'intégrité (FIM) pour surveiller toute modification suspecte dans /var/lib/docker.

Le Layer Poisoning nous rappelle que la sécurité d'un conteneur commence toujours par la sécurité de la machine qui l'héberge.
